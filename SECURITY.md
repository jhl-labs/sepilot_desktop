# 보안 정책 (Security Policy)

SEPilot Desktop 프로젝트의 보안을 유지하는 것은 매우 중요합니다. 보안 취약점을 발견하셨다면 아래 가이드를 따라주세요.

## 지원 버전 (Supported Versions)

현재 보안 업데이트를 제공하는 버전:

| 버전    | 보안 업데이트 지원 |
| ------- | ------------------ |
| 0.1.x   | :white_check_mark: |
| < 0.1.0 | :x:                |

## 보안 취약점 보고 (Reporting a Vulnerability)

### 공개하지 말아야 할 취약점

다음과 같은 보안 취약점을 발견했다면 **공개 이슈로 제보하지 마시고** 비공개로 보고해주세요:

- 인증/인가 우회
- 원격 코드 실행 (RCE)
- SQL Injection, XSS, CSRF 등 웹 취약점
- 중요 데이터 노출 (API 키, 비밀번호 등)
- 암호화 관련 취약점
- 서비스 거부 (DoS) 공격 가능성
- 기타 심각한 보안 이슈

### 보고 방법

1. **GitHub Security Advisory** (권장)
   - Repository의 "Security" 탭으로 이동
   - "Report a vulnerability" 클릭
   - 취약점 상세 정보 입력

2. **이메일 보고**
   - 프로젝트 메인테이너에게 직접 이메일 전송
   - 제목: `[SECURITY] 취약점 보고: [간단한 설명]`

### 보고 시 포함할 정보

보안 취약점 보고 시 다음 정보를 포함해주세요:

- **취약점 설명**: 무엇이 잘못되었는지 명확하게 설명
- **영향 범위**: 어떤 시스템/기능이 영향을 받는지
- **재현 방법**: 단계별 재현 절차
- **PoC (Proof of Concept)**: 가능하다면 개념 증명 코드 또는 스크린샷
- **심각도 평가**: Critical / High / Medium / Low
- **제안 해결책**: (선택사항) 어떻게 수정할 수 있을지 제안

## 보고 후 프로세스

1. **확인 (24-48시간 이내)**
   - 보고를 받았다는 확인 메시지 발송
   - 초기 평가 진행

2. **조사 및 검증 (1주일 이내)**
   - 취약점 재현 및 영향 범위 분석
   - 심각도 평가 및 우선순위 결정

3. **수정 및 패치**
   - 패치 개발 및 테스트
   - 보안 권고문 작성

4. **공개**
   - 패치 릴리스
   - 보안 권고문 공개
   - 발견자 크레딧 제공 (원하는 경우)

## 보안 모범 사례 (Security Best Practices)

SEPilot Desktop 사용자를 위한 보안 권장사항:

### API 키 관리

- **절대 하드코딩 금지**: 소스코드에 API 키를 직접 작성하지 마세요
- **환경변수 사용**: `.env` 파일 또는 설정 UI를 통해 관리
- **암호화 저장**: 앱은 설정을 암호화하여 저장합니다
- **정기 순환**: API 키를 정기적으로 갱신하세요

### GitHub OAuth 설정

- **Private Repository 사용**: 설정 동기화 시 반드시 Private 저장소 사용
- **Master Password 설정**: 강력한 마스터 패스워드 사용
- **최소 권한 원칙**: 필요한 최소한의 권한만 부여
- **토큰 관리**: Personal Access Token은 안전하게 보관

### 네트워크 보안

- **SSL/TLS 사용**: HTTPS API 엔드포인트만 사용
- **프록시 검증**: 신뢰할 수 있는 프록시만 설정
- **방화벽 규칙**: 필요한 포트만 허용

### 로컬 보안

- **OS 보안 업데이트**: 운영체제를 항상 최신 상태로 유지
- **안티바이러스**: 신뢰할 수 있는 보안 소프트웨어 사용
- **디스크 암호화**: 전체 디스크 암호화 권장 (FileVault, BitLocker 등)

## 알려진 제한사항 (Known Limitations)

### 암호화

- **Electron safeStorage**: 플랫폼별 암호화 사용
  - macOS: Keychain
  - Windows: DPAPI
  - Linux: Secret Service API 또는 libsecret
- **Fallback 암호화**: safeStorage 사용 불가 시 machine-specific 키 사용
  - 머신 고유 정보 기반 암호화
  - 다른 머신에서는 복호화 불가

### 데이터 저장

- **로컬 데이터베이스**: SQLite 데이터베이스는 앱 데이터 디렉토리에 저장
- **암호화되지 않은 데이터**: 채팅 히스토리는 평문으로 저장 (향후 암호화 예정)
- **GitHub 동기화**: config.json만 암호화하여 동기화

## 보안 개선 로드맵

다음 보안 개선사항이 계획되어 있습니다:

- [ ] 채팅 히스토리 암호화
- [ ] 데이터베이스 전체 암호화
- [ ] 2FA (Two-Factor Authentication) 지원
- [ ] 보안 감사 로그
- [ ] 자동 보안 스캔 CI/CD 통합
- [ ] 코드 서명 (Code Signing)

## 책임 있는 공개 (Responsible Disclosure)

우리는 책임 있는 보안 취약점 공개를 지지합니다:

1. **비공개 보고**: 취약점을 비공개로 먼저 보고
2. **협력**: 패치 개발 시 협력
3. **공개 대기**: 패치가 릴리스될 때까지 공개 대기
4. **크레딧**: 발견자에게 적절한 크레딧 제공
